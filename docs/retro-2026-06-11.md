## 主線一：導覽速度優化（每次點按鈕等 3-5 秒）

### 問題根源分析

用戶反映每按一個按鈕都要等 3–5 秒。逐步診斷：

1. **第一層誤解**：以為是 Turbopack dev 模式每次重新編譯 → 其實 Turbopack 第一次造訪後就快取，之後不重新編譯
2. **真正根因 A**：`<a href>` 標籤造成整頁刷新（full page reload），而不是 Next.js 的 client-side routing
3. **真正根因 B**：NextAuth 使用 `strategy: "database"`，每次 `auth()` 都去 Supabase sessions 表 SELECT 一次（~200ms round-trip），加上 DB 查詢，每次導覽總共 2-3 秒

### 解法

**修一：`<a>` → `<Link>`**（client-side routing）
- `TrackButton.tsx`：去收藏區按鈕
- `RecommendationList.tsx`：看板連結、求職意圖編輯
- `SavedJobList.tsx`：「投遞準備」按鈕
- `board/[applicationId]/page.tsx`：「← 回求職追蹤」

**修二：JWT session 策略**（消除每次的 auth DB 查詢）
```typescript
// auth.ts 改動
session: { strategy: "jwt" }  // 從 "database" 改過來

callbacks: {
  jwt({ token, user }) { if (user) token.id = user.id; return token; },
  session({ session, token }) { session.user.id = token.id as string; return session; }
}
```
效果：`auth()` 改為從 JWT cookie 解碼，0 次 DB round-trip，省掉每次導覽 ~200ms。

**修三：`loading.tsx` 骨架畫面**（改善體感速度）

在 5 個主要路由加入骨架畫面，讓用戶點擊後立即看到內容框架，不再看到空白等待：
- `app/dashboard/loading.tsx`
- `app/saved/loading.tsx`
- `app/board/loading.tsx`
- `app/board/[applicationId]/loading.tsx`
- `app/profile/loading.tsx`

### 注意事項
切換到 JWT strategy 後，舊的 database session 全部失效，用戶需要重新登入一次。

## 主線二：Railway 爬蟲 Crash 修復

### 症狀
Railway 服務顯示 `● Crashed`，log 顯示：
```
psycopg2.OperationalError: connection to server at
"db.oilzqorglwmrysahrgnf.supabase.co" port 5432 failed:
Network is unreachable
```

### 根因
`DIRECT_URL` 設的是 Supabase 直連 hostname（`db.xxx.supabase.co`），Railway 的 SFO region 網路解析出 **IPv6 位址**（`2406:...`），但 Railway 容器**不支援 IPv6**，連線直接失敗。

### 解法
把 Railway 環境變數 `DIRECT_URL` 從直連 hostname 換成 **Supabase pooler hostname**（IPv4）：

```
舊：postgresql://postgres:...@db.oilzqorglwmrysahrgnf.supabase.co:5432/postgres
新：postgresql://postgres.oilzqorglwmrysahrgnf:...@aws-1-ap-southeast-2.pooler.supabase.com:5432/postgres
```

驗證方式：本機 `psycopg2.connect(pooler_url)` 回傳 `(1,)` 確認通。重新 deploy 後狀態改回 `● Online`。

### 爬蟲排程
Cron：Railway 每天 **13:00 UTC（台灣 21:00）** 執行爬蟲
Matching：Vercel cron 每天 **13:30 UTC（台灣 21:30）** 觸發 `/api/cron/daily`

手動測試流程（本機）：
```bash
# Step 1：爬職缺
cd worker/crawler
DIRECT_URL="postgresql://..." python3 run.py

# Step 2：跑 matching（dev server 需跑著）
curl -H "Authorization: Bearer $CRON_SECRET" http://localhost:3000/api/cron/daily
```

## 主線三：Vercel 正式部署

### 背景
App 原本只在 localhost 跑，導致：
- Vercel cron（`/api/cron/daily`）不會自動執行
- 爬蟲抓到的職缺不會自動出現在網站
- 導覽速度慢（dev 模式 Turbopack 懶編譯）

### 部署流程與踩坑

**坑一：Vercel build 失敗**

錯誤：`Command "pnpm run build" exited with 1`

原因：Vercel 安裝的是全新 node_modules，沒有 Prisma client（本機有是因為已經 `prisma generate` 過）。

修法：`package.json` build script 加 `prisma generate`：
```json
"build": "prisma generate && next build"
```

**坑二：GitHub 連線失敗**

`vercel link` 時出現 `Failed to connect annie1010123/job-tool to project`。GitHub App 沒有安裝到 repo，因此無法自動部署。

處理方式：改為手動 CLI 部署，每次更新跑 `vercel --prod --yes` 即可（約 2-3 分鐘）。

**坑三：NEXTAUTH_URL 沒設定**

手機連不上的原因之一。Vercel 環境沒有 `NEXTAUTH_URL`，auth redirect 跑回 localhost。

修法：
```bash
echo "https://job-tool-three.vercel.app" | vercel env add NEXTAUTH_URL production --yes
```

### 環境變數清單（Vercel Production）
共 15 個：`NEXTAUTH_SECRET`, `NEXTAUTH_URL`, `GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET`, `DATABASE_URL`, `DIRECT_URL`, `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY`, `GEMINI_API_KEY`, `GROQ_API_KEY`, `RESEND_API_KEY`, `EMAIL_FROM`, `CRON_SECRET`, `AUTH_TRUST_HOST`

### Google OAuth 更新
在 Google Cloud Console → OAuth Client → 新增：
- Authorized JS origins：`https://job-tool-three.vercel.app`
- Authorized redirect URIs：`https://job-tool-three.vercel.app/api/auth/callback/google`

**https://job-tool-three.vercel.app**
