# 設計文件：面試準備 — 核心題 + 答案教練 + 故事庫

> status: draft（待使用者審閱）
> date: 2026-06-19
> 對應 CLAUDE.md：北極星「每週 AI 面試準備使用次數」、Aha moment「看到題目標註『根據你上次面試經驗延伸』」

---

## 1. 背景與問題

JobPilot 真正的差異化不在職缺聚合（競品如 Jobbo 已做得很多），而在「**從每次面試中學習、越用越強的面試準備**」。

現況（已建骨架）：
- `ai-questions/route.ts`：依 JD + 公司類型出 10–12 題，並把 `QuestionBank` 加權進去。
- `lib/review/analyze.ts`：面試**後**上傳錄音/手填 → 萃取 Q&A、評品質、給建議。

缺口（使用者訪談收斂）：
1. **題目不夠精準**：LLM 只拿到 JD，產出像通用猜測。
2. **沒有「練習當下」的答案教練**：使用者想「我給一個答案 → AI 幫我改得更好 → 再改一版」的一來一回迭代，現有復盤是「事後回顧」，不是「事前磨稿」。
3. **準備綁定職缺**：所有面試準備都掛在某個 `Application` 下，使用者沒有職缺就無法練；通用題（自我介紹、最大挫折…）無處可練。

**競爭基準線**：使用者可以把 JD 貼到 ChatGPT 拿到題目。JobPilot 必須贏過這條線——靠的是 ChatGPT 拿不到的東西：**使用者的履歷/經歷**，以及**會累積、可重複用的個人答案資產**。

---

## 2. 目標與非目標

### 目標
- 建立一個**互動式答案教練引擎**：給定題目 + 使用者答案 → 產出評分 + 具體可執行的修改建議 → 支援多次迭代。
- 提供**核心題練習**（萬年通用題），不依賴任何職缺，第一天就能用。
- 把磨好的答案沉澱為**可重複用的個人故事庫**，並在 JD 情境中被引用。
- 把「填個人資料」的摩擦降到接近零：以履歷自動帶入 + 練習即填寫取代表單。

### 非目標（YAGNI / Not Doing）
- ❌ 真實考古題爬取（Glassdoor / 面試趣）— 法律風險高、覆蓋差。改以「跨使用者題庫累積」當未來的網路效應，冷啟動前不做。
- ❌ 對話追問型面試官（②選項）/ 開口說語音練習（③選項）— 留待後續 Phase，本 spec 不含。
- ❌ 答案的完整版本歷史 / diff — v1 只存「最新答案 + 最近一次回饋 + 分數」。
- ❌ 模擬整場面試。

### 排序決策（ICE）
答案教練引擎（①）ICE 最高且是 ②③ 的地基 → 先做。核心題（固定清單）當作引擎的**第一個載體**，因為它繞過「題目精準」難題、無前置依賴。JD 精準題列為 Phase 2。

## 3. 使用者與 JTBD

- **目標用戶**：正在找實習/工作的學生與初階工程師。
- **JTBD**：當我要準備面試時，我想要一個能根據我自己的答案、一步步把它磨得更好、而且記得我練過什麼的教練，以便我每一場面試都更有把握、且不用每次從零開始。

## 4. 核心概念

| 概念 | 定義 | 既有對應 |
|------|------|----------|
| **核心題** | 萬年通用、跨公司都會問的題（自我介紹、最大挫折、最大困難、解決問題的經驗…）。固定策展清單。 | 新增（程式內 seed 清單） |
| **JD 精準題** | 綁定某職缺、由 JD × 履歷落差生成的題。 | `Application.aiQuestions`（Phase 2 強化） |
| **答案教練引擎** | 給定（題目 + 使用者答案 [+ 履歷/經歷 context]）→ 評分 + 結構化修改建議；可迭代。 | 新增（複用 Groq 與 `analyze.ts` 的 prompt 風格） |
| **個人故事庫** | 使用者磨好的答案集合，可重複用、會演進。 | 擴充 `QuestionBank` |
| **素材** | 原始工作/專案經歷、履歷。 | `WorkExperience`、`Resume` |

**層次**：素材（原始）→ 練習（答案教練）→ 故事庫（成品）。故事庫是「成品」，個人資料是「素材」。

## 5. 功能範圍

### Phase 1（本 spec 主體）
1. **核心題練習頁**：列出策展核心題；每題顯示準備狀態（未練 / 已備）。
2. **答案教練迴圈**：
   - 進入一題 → 若使用者有履歷/經歷，AI 先生**初稿**（不給空白框）。
   - 使用者編輯/輸入答案 → 送出 → AI 回傳：分數（0–100 或等級）+ 3 條具體修改建議（指出哪句空泛、STAR 缺哪段、缺什麼佐證）。
   - 使用者改一版再送 → 再回饋。迭代直到滿意 → 「存成我的故事」。
3. **個人故事庫**：列出所有已存答案，可重看、再練、編輯。
4. **履歷自動帶入**：練習 context 取自既有 `Resume` 解析與 `WorkExperience`，無需手填表單。

### Phase 2（後續，本 spec 只描述介面契約）
- **JD 精準題**：兩步驟（抽 JD 結構 → JD × 履歷落差比對 → 出落差題/查證題），複用同一答案教練引擎，並可調用 Phase 1 的故事。
- 在職缺面試準備分頁中**引用**核心題（顯示準備度、可跳回故事庫）。

## 6. 資訊架構（IA）

新增頂層導覽「**面試準備**」（理由：核心題不綁職缺、且北極星指標不該被埋在三層點擊下）。

```
主頁
找工作
求職追蹤
  └ 職缺 → 〔面試準備分頁〕= JD 精準題（Phase 2）；引用↓故事庫，不擁有
面試準備   ← 新增頂層 /interview
  ├ 核心題練習（自我介紹、最大挫折…）+ 答案教練
  └ 我的故事庫（磨好的答案，唯一真身，可複用）
個人資料   = WorkExperience + 履歷（素材層，維持現狀）
```

**單一真身 + 多處引用**：故事庫是唯一儲存處；職缺頁只「引用顯示」並深連回故事庫，不複製答案。

## 7. 互動設計（讓「連貫」被感受到）

原則：**不靠解釋，靠在重用的瞬間讓使用者看到自己的努力回來幫他。**

1. **重用瞬間（最強）**：Phase 2 職缺準備時顯示「這場也會問〔最大挫折〕✅ 你已備好 →〔套用〕〔針對這家公司微調〕」，點下去帶入既有答案。
2. **視覺標籤**：♻️ 通用題 vs 📍 這家公司題；同一張 ♻️ 卡片在「面試準備」與職缺頁長一樣 = 同一個東西。
3. **帶著走的準備度**：職缺頁頂顯示「核心題 4/6 已備｜這家公司 0/5」；4/6 跨職缺一致 → 使用者領悟「核心準備跟著我」。
4. **雙向連結 + 微文案**：職缺核心題卡標「來自你的故事庫 →」；重用當下文案「你練好的〔最大挫折〕已帶進這場 ✓」。
5. **期待鉤子**：第一次練核心題結尾提示「這些答案存起來了，之後每場面試都用得上」。

**時序**：連貫感在**第二次互動**才被兌現（第一次練時尚無職缺可重用）→ 先承諾（招5），後兌現（招1）。

## 8. 答案教練引擎

### 流程
```
題目 (+履歷/經歷 context)
  → [可選] 生成初稿
  → 使用者答案 v_n
  → 教練評估：score + 結構化建議[]  (Groq llama-3.3-70b，temp 低)
  → 使用者修改 → v_(n+1) → 再評估
  → 滿意 → 存入故事庫（QuestionBank）
```

### 評估輸出（JSON，沿用 `analyze.ts` 風格）
```jsonc
{
  "score": 0-100,
  "strengths": ["..."],            // 1-2 條
  "improvements": [                 // 2-3 條，具體可執行
    { "issue": "結尾沒有量化成果", "suggestion": "補上一句數字結果，例如…" }
  ],
  "starCoverage": { "S": true, "T": true, "A": false, "R": false }, // 行為題適用
  "revisedExample": "（可選）示範改寫一段"
}
```

### Prompt 要點
- system：資深面試官 + 教練；標準＝具體性/STAR 完整/有量化佐證/真實口吻；繁中。
- user：題目 + 題型 + 使用者答案 + 履歷/經歷摘要（截斷）。
- 行為題要求檢查 STAR 四要素；動機題檢查與該角色/產業的連結。

## 9. 資料模型

**策略：擴充既有 `QuestionBank` 當故事庫**，避免新增重複模型。

```prisma
model QuestionBank {
  // ── 既有欄位（保留）──
  id                  String
  userId              String
  question            String
  answer              String?  @db.Text   // 最新一版答案
  category            String   @default("other")
  sourceApplicationId String?
  userPerformance     String?
  frequency           Int      @default(1)
  createdAt           DateTime
  updatedAt           DateTime

  // ── 新增欄位 ──
  isCore        Boolean  @default(false)   // 是否為核心通用題
  coreKey       String?                    // 核心題的穩定識別（self_intro / biggest_failure …），同 user 同 coreKey 唯一
  answerScore   Int?                       // 最近一次教練評分
  lastCoaching  Json?                      // 最近一次教練回饋（strengths/improvements…）
  answeredAt    DateTime?

  @@unique([userId, coreKey])              // 確保核心題每人單一真身
}
```

- **核心題清單**：以程式內 seed 常數定義（`coreKey`、label、category、出題/初稿提示），不建 DB 表。使用者首次練某核心題時建立對應 `QuestionBank` 列（`isCore=true`）。
- **故事庫** = `QuestionBank` 中該 user 的列（核心題 + 來自復盤的真實題）。
- **與復盤整併**：`InterviewReview` 萃取的 Q&A 持續回灌 `QuestionBank`（沿用現有邏輯）；這些列 `isCore=false`、帶 `sourceApplicationId`。
- **遷移**：新增欄位皆 nullable / 有預設，對既有資料安全。

## 11. 錯誤處理與邊界

- LLM 回傳非合法 JSON → 重試一次，再失敗回 fallback（沿用 `analyze.ts` try/catch 模式）。
- 使用者無履歷/經歷 → 跳過初稿生成，給空白框 + 一兩個輕量 scaffolding 提示（不擋住練習）。
- 空答案 / 過短答案 → 前端擋，提示先寫幾句。
- 教練 API per-user rate limit（沿用 CLAUDE.md 既有規範），設每日上限控成本。
- 所有 API 需 auth；故事庫資料受 Supabase RLS，使用者只能存取自己的。

## 12. 測試

- `lib/interview/coach.ts`：用 recorded LLM fixture 測（沿用 CLAUDE.md「LLM 用 fixture」規範）——含「好答案高分 / 空泛答案低分且 improvements 非空」兩案。
- core-questions seed：基本結構驗證（每題有 coreKey/label/category）。
- API integration：未登入 401；存答案後可在故事庫讀回；重複練同一 coreKey 不產生重複列（`@@unique` 驗證）。

## 14. 風險與緩解

| 風險 | 緩解 |
|------|------|
| 教練回饋空泛（淪為 ChatGPT） | prompt 強制「指名哪一句、缺哪個 STAR 段、給可貼上的改寫」；fixture 測試把關。 |
| 初稿讓使用者照抄、失去個人性 | 初稿定位為「起點」，UI 文案鼓勵改寫；評分對「太通用」扣分。 |
| 連貫感無感 | 第 7 節的重用瞬間 + 視覺標籤 + 準備度。 |
| 成本 | 每日上限 + 低 temp + 答案長度截斷。 |

## 16. 最終定案（2026-06-20，UI 已 mockup 定稿）

> 互動細節以 mockup 為準：`docs/mockups/interview-prep.html`（單檔 HTML，可瀏覽器開）。

### 16.1 資訊架構（合併）
- **單一「面試準備」頁**，頂部視角切換 **[核心題 ｜ 常被問到]**（不再有獨立「故事庫/題庫」頁，避免重複）。
- 進度條（核心題準備度 X/Y）＋北極星（本週練習次數）放最上。

### 16.2 核心題視角
- **Notion 摺疊清單**：題目當主角；右側狀態為「✓ N 版本」（已備）或「現在練」（未練）。
- **已答**：點 ▶ 展開 → **唯讀**看答案（透明框）＋版本切換／＋版本 ＋ 右上「編輯」。展開內容**用滿整列寬度**。
- **未答**：不可展開，整列點擊 →「現在練」直接進教練彈框。
- 視角列右側固定一顆「**＋ 新增題目**」（自訂題：輸入題目文字＋題型 → 加入清單）。

### 16.3 答案教練彈框（兩欄）
- 由「編輯／現在練」開啟。**左欄＝我的答案**（版本 tabs＋AI 初稿＋儲存＋「請教練評估」，評估後按鈕變「重新評估」）；**右欄＝教練回饋**（空狀態 → 評估後：**評等三級（待加強／可／優秀）**＋做得好＋可以更好（每點附「示範」句）＋結構檢查）。
- **編輯題目／刪除**：放在彈框**標題右側**（「刪除題目」紅字）。清單上不再放編輯/刪除。
- 內容過長時欄位**內部捲動**，彈框本身不放大；彈框上下置中（`margin:auto`）。

### 16.4 版本
- 自我介紹預設 `30秒`／`1分鐘`＋`＋新增版本`（自訂名稱）；其他題預設 1 版，展開後左上有「＋版本」可加。

### 16.5 常被問到視角
- 由**面試後復盤勾選「這題被問到了」** → 累積 `QuestionBank.frequency` → 此視角依次數排序，顯示「被問 N 次・哪些公司」，未答的給「現在練」。= 個人化考古題（冷啟動時空狀態）。

### 16.6 技術 / 商業決策
- **模型分工**：答案教練用 **Claude API（Haiku/Sonnet 級起跳，品質＝差異化）**；104 配對等廉價任務維持 Groq llama。
- **Freemium**：AI 教練免費 **5 次** → 之後付費解鎖；**資料管理（題庫/版本/追蹤）永久免費**。因有次數上限，教練成本封頂，故負擔得起好模型；防爆＝每人每日上限＋prompt caching。
- **定位**：不是「另一個 AI 寫手」，而是「**記得你、越用越強的求職大腦**」（護城河＝累積＋管理＋融入求職流程，而非 AI 呼叫本身）。

### 16.7 配色
- 暖色 token：背景 `#f1efe8`、卡片白、暖灰字 `#888780/#1a1a18`、暖邊 `#e8e6df`、米色按鈕 `#e6e0d2`（不用紫色）。

### 16.8 開放問題（拍板）
- 自訂題：**v1 做**（清單有「＋新增題目」、可編輯/刪除）。
- 評分呈現：**三級（待加強／可／優秀）**，內部可另存分數。
