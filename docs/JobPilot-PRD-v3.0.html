1Background

問題陳述

台灣的求職者日常是這樣的：打開 104 搜一輪、切到 LinkedIn 搜一輪、再去 CakeResume 看看，然後把有興趣的職缺複製到 Notion 或 Excel 裡追蹤。每投一間公司，就要重新寫一封推薦信、重新準備面試題目。投了 15 間、面了 8 間，但第 9 間的準備品質跟第 1 間一模一樣。

問題不是缺工具，而是工具之間不連通。求職的每個環節——搜尋、篩選、投遞、準備、面試、復盤——都有各自的解法，但沒有任何一個產品把它們串起來，讓每次面試的經驗累積下來。

市場現況

平台做得好的缺什麼
104 人力銀行職缺量最大、台灣求職者首選搜尋靠關鍵字、沒有語意理解、推薦信要自己寫
LinkedIn海外職缺、人脈網絡台灣職缺不完整、推薦演算法偏社交互動而非職缺匹配
CakeResume履歷設計、新創職缺多沒有面試準備功能、追蹤要靠外部工具
Notion / Excel彈性高、可自訂純手動、沒有 AI、投遞狀態靠自己記
ChatGPT可以問面試題、寫推薦信沒有記憶、每次對話都從零開始、不知道你投了哪些公司

機會

沒有一個產品同時做到「AI 語意推薦職缺」+「自動生成推薦信」+「面試準備會根據歷史經驗進化」。這三件事串起來的價值遠大於各自獨立存在。一個記得你所有求職歷程的 AI 助手，可以讓每一次面試都比上一次更好。

當我正在積極求職、每天要在多個平台切換找職缺、改履歷、準備面試時，我想要一個記得我所有資料、會從每次面試中學習的 AI 助手，這樣我就能把時間花在真正重要的面試準備上，而不是重複的行政工作。

5Product Strategy

產品定位

JobPilot 不是另一個求職平台，也不是通用 AI 助手。它是一個專為積極求職者設計的 AI 求職教練，核心差異化在於：面試經驗的累積。每一次面試的復盤都會讓下一次的 AI 準備更精準。

與現有工具的定位差異

工具核心價值JobPilot 不同在哪
104 / LinkedIn / Cake職缺資訊來源JobPilot 整合多平台 + AI 語意匹配，不取代它們而是串起來
Notion / Excel手動追蹤投遞自動化追蹤 + 久沒回音警示 + 漏斗轉換率
ChatGPT通用問答結構化題庫 + 歷史經驗累積，不是每次從零開始
Jobbo多平台整合 + 滑卡篩選JobPilot 多了面試準備 + 復盤 + AI 進化的完整後半段

6Success Metrics

North Star Metric

每週 AI 面試準備使用次數
為什麼選這個而不是 DAU 或投遞數：它反映核心差異化（AI 從面試經驗中進化），具有累積效應（用越多越精準），而且無法被其他工具取代。類比 Figma 選「跨職能協作次數」而非「設計檔數量」。

MVP 階段數據不足以衡量面試準備，先用「推薦職缺收藏率」作為先行指標。產品進入成長期後切換。

Hero Metrics

指標定義目標值為什麼重要
AI 面試準備使用率有投遞紀錄的用戶中，使用過 AI 面試準備的比例≥ 40%驗證核心功能的被需要程度

7MVP Scope & Rationale

MVP 要驗證什麼

MVP 的目標不是做出完整產品，而是回答三個問題：

  1. AI 語意推薦有沒有用？（H1）— 用戶會不會收藏推薦的職缺
  2. 每日推薦能不能帶來回訪？（H2）— Email 點擊率和留存
  3. 自動追蹤比手動好用嗎？（H3）— 用戶會不會真的用 Kanban 追蹤

MVP 包含的功能

功能做什麼驗證哪個假設
Landing Page + 30 秒試玩訪客輸入求職方向，即時看到 AI 推薦的 Top 5 職缺降低註冊門檻，展示核心功能
Email Magic Link 登入不需要設密碼減少註冊摩擦
Onboarding（求職關鍵字設定）用戶輸入想找的工作方向 + LLM 擴展關鍵字H1 前置條件
104 爬蟲 + Embedding Match每日爬取職缺、語意匹配、產出推薦清單H1
每日 Email 推薦新推薦以 Email 送達 + 點擊追蹤H2
Dashboard問候語 + 漏斗統計 + 待處理 + 最近動態H2, H3
找工作頁（AI 推薦 + 收藏）瀏覽推薦、收藏、篩選、搜尋H1
求職追蹤（清單 + Kanban）投遞狀態管理 + 漏斗轉換率 + 久沒回音警示H3

MVP 刻意不做的

砍掉什麼為什麼
履歷上傳 + 解析MVP 的推薦靠關鍵字意圖匹配即可，履歷是推薦信生成的前置條件，跟推薦信一起放 V2
推薦信 AI 生成先確認推薦有用、用戶會回來，再加生成功能
AI 面試準備面試準備是核心差異化，但需要用戶先有投遞數據才有意義。MVP 先累積數據
面試錄音 + 復盤依賴 Whisper API 成本較高，且需要 H5 假設成立才值得投入
題庫自動進化需要面試復盤功能先上線，是 V2 的後段
完整個人檔案MVP 只需要求職關鍵字，完整的工作/專案/技能管理放 V2

8V2 Roadmap & Rationale

V2 啟動條件

當 MVP 數據顯示以下信號時，進入 V2 開發：

  • 推薦職缺收藏率 ≥ 40%（H1 成立 — AI 推薦有用）
  • Email 點擊率 ≥ 15%（H2 成立 — 有回訪動力）
  • 至少 5 位用戶有 3+ 筆投遞記錄（有足夠數據支撐面試準備）

V2 功能排序與理由

順序功能為什麼這個順序驗證假設
V2-1推薦信 AI 生成投遞流程的最大摩擦點，且 MVP 數據已確認用戶會投遞。降低「收藏到投遞」的轉換門檻H4
V2-2個人檔案（工作經歷 + 專案 + 技能）推薦信品質取決於對用戶背景的理解深度，完整檔案 = 更好的推薦信H4 支援
V2-3AI 面試準備（基礎版）有投遞數據後，可以根據 JD + 履歷出題。這是北極星指標的核心功能North Star
V2-4面試錄音上傳 + AI 復盤面試準備上線後，用戶需要一個閉環：準備 → 面試 → 復盤 → 更好的準備H5
V2-5手動面試記錄錄音復盤的低門檻替代方案，確保面試後太累的用戶也能回饋H5 fallback
V2-6題庫自動進化復盤數據自動回饋到 AI 出題，實現「每次面試都讓你更強」的核心承諾North Star

9.1b 優先級定義與排序理由

優先級定義判斷標準
P0沒有就不能用拿掉這個功能，產品的核心路徑會斷掉
P1核心體驗大幅提升有了之後用戶體驗有質的飛躍，但沒有也能完成基本任務
P2有了更好提升便利性或資訊密度，但不影響核心功能

9.1c 排序理由

功能優先級為什麼這樣排
F0 Landing + 試玩P0用戶進來的第一個接觸點，沒有就沒有轉換
F1 登入P0不能登入就沒有後續所有功能
F2 OnboardingP0不設定關鍵字就無法做語意推薦
F3 爬蟲 + 推薦P0整個產品的資料來源，沒有職缺數據其他功能都是空的
F5 找工作頁P0用戶看推薦、收藏、篩選的主要介面
F6 求職追蹤P0投遞後的追蹤和管理，沒有的話跟 Notion 沒差別
F7 推薦信P1大幅降低「收藏→投遞」的轉換摩擦，但用戶可以先手動寫
F9 AI 面試準備P1北極星指標的核心功能，但需要先有投遞數據才有意義
F11 手動記錄P1比錄音復盤更重要 — 是確保「面試→復盤」閉環不斷的低成本方案
F4 DashboardP2有找工作頁和追蹤頁就能操作，Dashboard 是資訊匯總，不是必要路徑
F8 個人檔案P2提升推薦信品質，但基本版只用履歷解析就能生成
F10 錄音復盤P2Whisper 成本高且依賴 H5 假設成立，手動記錄是更安全的起點
F12 題庫進化P2需要先累積復盤數據，是復盤功能的延伸而非獨立功能

9.2 F0 Landing Page + 試玩

觸發條件

任何訪客進入首頁（不需登入）

頁面結構

  1. Hero 區：左文右圖佈局。標語強調好處、右側放產品截圖、雙 CTA + 信任點
  2. 痛點區：列出求職者日常的具體痛苦
  3. 三步驟解法：01 AI 推薦 / 02 投遞追蹤 / 03 面試 AI 進化，每步配產品截圖
  4. 30 秒試玩區：輸入求職方向 → 即時顯示 AI 推薦職缺
  5. 數據 banner + Final CTA + Footer

試玩職缺推薦規格

  • 輸入：一句話描述求職方向（input，placeholder 提示「例如：PM 實習 台北」）
  • 後端：將輸入轉成 embedding，對已爬取的 JD 做 pgvector similarity search
  • 輸出：Top 5 最匹配職缺（職稱 + 公司名 + 相符度 %）
  • 限制：Rate limit 5 次 / 小時 / IP
  • CTA：「想收藏這些職缺？免費註冊」

9.3 F3 爬蟲 + 推薦 + Email

爬蟲機制

  • 每日 cron 觸發，爬取 104 職缺（根據用戶 JobIntent 的關鍵字）
  • 使用 Python curl_cffi 繞過 Cloudflare，搭配 Playwright fallback
  • 爬取欄位：職稱、公司名、地點、薪資、技能需求、職缺描述、招募活動度
  • 自動去重（by externalUrl unique constraint）

語意匹配

  • 職缺 JD 和用戶履歷 / 意圖分別轉成 768 維 embedding（pgvector）
  • finalScore = 0.6 * resumeScore + 0.4 * intentScore
  • 推薦清單按 finalScore 排序，保留近三天數據

Email 推薦

  • 每日一封，包含 Top 5 推薦 + 點擊追蹤 token
  • 技術：Resend + React Email 模板

9.4 F6 求職追蹤

清單視圖

  • 頂部漏斗：投遞中 → 面試中 → Offer，顯示轉換率
  • 分群顯示：投遞中 / 未投遞 / 錄取 / 封存
  • 每列顯示：公司名、職缺、狀態標籤、天數
  • 久沒回音警示：投遞超過 14 天未更新狀態，顯示警示標籤 + 封存按鈕

看板視圖

  • 四欄：未投遞 / 投遞中 / 面試中 / 結果
  • 卡片可拖曳切換狀態

9.5 F7 推薦信 AI 生成（V2）

觸發條件

用戶在 Application 詳情頁點擊「推薦信」tab → 點擊「生成推薦信」

正常流程

  1. 前端 POST /api/cover-letter/generate with { applicationId }
  2. 後端讀取用戶 Resume + 個人檔案（工作經歷、專案經歷）
  3. 後端讀取 JD（title, companyName, skills, description）
  4. 組裝 prompt → Groq LLM（llama-3.3-70b, temp 0.7）
  5. 回傳推薦信文字（300-500 字），專業正式語氣
  6. 前端顯示在可編輯 textarea，用戶可自行調整風格或內容
  7. 提供「複製」和「重新生成」按鈕

例外流程

  • 履歷未上傳 → 400 +「請先上傳履歷」
  • LLM 失敗 → 500 + retry 1 次
  • 回應超時（>15s）→ 前端顯示 timeout 提示

9.6 F10 面試錄音 + AI 復盤（V2）

觸發條件

用戶在 Application 詳情頁點擊「面試復盤」tab

錄音路徑

  1. 上傳音檔（mp3/m4a/wav, max 50MB）→ Supabase Storage
  2. Whisper API 轉文字（$0.006/min，中英混合支援）
  3. Groq LLM 萃取 Q&A：每題含 question, answer_summary, quality（good/ok/needs_improvement）, improvement_tip, category
  4. 生成 overallFeedback（整體摘要 + 3 項改善重點）
  5. 自動寫入 QuestionBank

手動路徑（F11）

  1. 用戶填寫多題 Q&A（題目 + 回答 + 自評）
  2. 進入同一個 AI 分析 pipeline（跳過 Whisper）
  3. 自動寫入 QuestionBank

14Risks & Assumptions

風險矩陣

風險嚴重度預防措施
104 爬蟲被 Cloudflare 擋高curl_cffi 模擬真實瀏覽器 + Playwright fallback + parse-jd API 可解析單一 URL 作為退路。Day 1 就建立爬蟲健康監控
面試後太累不做復盤高手動快速問卷 fallback（1 分鐘完成）+ 48 小時推播提醒
Scope creep 做不完高嚴格 MVP / V2 分界 + Not Doing List + 每週範圍檢查
LLM 成本超預算中使用 Groq（免費額度）+ 設每日呼叫上限 + 護欄指標 ≤ $2/user/month
語意推薦不夠精準中先用收藏率做 proxy metric，不準就調整 embedding model 或 scoring 權重
ChatGPT 加入結構化記憶功能中深耕結構化題庫 + 全流程整合（推薦 → 投遞 → 準備 → 復盤）是 ChatGPT 短期不會做的垂直場景
