# JobPilot 作品集 — 完整大綱

> 目標：PM 實習面試，展示從問題發現到產品迭代的完整思考過程
> 呈現方式：Notion 頁面（完整版）+ 簡報 12 頁（精華版）+ Live Demo

---

## 呈現方式建議

| 方式 | 適合場景 | 建議 |
|------|---------|------|
| **Notion 頁面** | 投履歷時附連結，面試官自己看 | ✅ 主力，深淺兩層 |
| **簡報（12 頁）** | 面試中 8-10 分鐘簡報 | ✅ 必備，從 Notion 濃縮 |
| **Live Demo** | 面試中現場展示 | ✅ 加分，準備 3 個場景 |

核心原則：Notion 是完整版（面試官自己翻），簡報是精華版（你講給他聽）。

---

## 故事線：7 幕結構

### 第一幕：Why — 問題從哪裡來（展示：問題發現能力）

**講什麼：**
- 你自己找實習的真實痛苦（不是假設的，是親身經歷的）
- 觀察到的具體行為：每天逛 104 一小時、用 Line 記事本存職缺、ChatGPT 每次都要重新介紹自己
- 一句話定義問題：「求職最大的浪費不是時間，是每次面試的經驗沒有被留下來」

**交付物：**
- 當初的求職工具截圖（Excel / Notion / Line 記事本 — 越亂越好，證明痛點真實）
- 問題陳述（1 句話）

**面試官想聽你說：**
> 「這不是我想像出來的問題，是我自己在找實習時每天都在經歷的。我發現我投了 10 間，但第 11 間的面試準備品質並沒有比第 1 間好。」

---

### 第二幕：Who — 誰有這個問題（展示：用戶研究能力）

**講什麼：**
- 做了什麼來驗證「不只是我有這個問題」
- 4 位受訪者的訪談
- 從訪談中發現的 insight（4/4 都提到面試經驗留不住）

**交付物：**
- Persona 卡片（P1 積極求職者 Alex / P2 被動觀察者 Mia / P3 工程主管 Reviewer）
- 訪談問題清單 + 關鍵語錄
- 訪談洞見彙整表
- User Journey Map（9 階段 × 行為/情緒/痛點/機會點）

**怎麼講思考過程：**
> 「我的 Persona 切分不是用年齡性別，而是用『求職動機和行為模式』。因為 25 歲積極求職和 28 歲被動觀察的人，需要的功能完全不同。」

> 「Journey Map 不是我憑空畫的。我先訪談，把每個受訪者的求職流程拆成步驟，發現大家都會經過 9 個階段。然後把痛點標上去，發現階段 3（寫推薦信）和階段 7-8（面試後復盤）是情緒最低點。」

**呈現方式：**
- Journey Map 用橫向表格，每階段一欄
- 情緒曲線用 emoji 或顏色（紅 = 最痛）
- 關鍵痛點旁邊寫「→ 對應的 HMW 問題」

---

### 第三幕：What — 要做什麼、不做什麼（展示：優先級判斷 + 取捨能力）

**講什麼：**
- JTBD 分析（功能性 / 情感性 / 社交性）
- 8 個 HMW 問題 → 用影響力 × 可行性打分排序
- 為什麼選這 3 個做 MVP，為什麼其他的不做
- Not Doing List

**交付物：**
- JTBD 分析表 + 深挖五問
- 痛點彙整表（8 個痛點 × 影響程度 × 頻率）
- HMW 問題轉化（8 個 HMW）
- Opportunity 排序表（Tier 1/2/3）
- Impact / Effort Matrix 四象限圖
- Positioning 定位表（April Dunford 五要素）
- 競品差異矩陣（104 / LinkedIn / ChatGPT / Notion / JobPilot）
- MVP 功能清單 vs Not Doing List

**面試官想聽你說：**
> 「我有 8 個 HMW 問題，用三個維度打分：用戶影響、展示價值、可行性。最後選出 Tier 1 的三個。」

> 「Not Doing List 跟做什麼一樣重要。我明確不做自動投遞（法律風險）、不做即時 AI 提示（道德問題）、不做付費系統（作品集不需要）。」

---

### 第四幕：User Story + 產品路線圖（展示：規劃能力）

**交付物：**
- User Stories 完整清單（6+ 個，每個含驗收標準）→ 見 `user-stories.md`
- Product Roadmap 時間軸（5 個 Phase × 8 週）

**排序邏輯：**

```
Phase 0（W0）      Phase 1（W1-2）    Phase 2（W3-4）     Phase 3（W5-7）      Phase 4（W8）
Landing Page       職缺收藏+投遞      推薦信 AI 生成      錄音復盤+AI 進化      UI 優化+部署
US-006            US-001, US-002     US-003             US-004, US-005       統一設計+README
```

**面試官想聽你說：**
> 「Phase 0 先做 Landing Page，因為 Reviewer 是我最重要的受眾之一。Phase 1 串起核心流程。Phase 2 做 RICE 最高的推薦信。Phase 3 做差異化最大的 AI 進化。」

> 「User Story 的排序按照用戶流程：先有職缺 → 收藏 → 投遞 → 面試 → AI 進化。每個 story 的驗收標準都可以直接拿來寫測試。」

---

### 第五幕：Wireframe + 設計決策（展示：設計思維）

**交付物：**
- 5 個核心頁面的 Wireframe → 見 `wireframes/` 資料夾
  1. Landing Page — hero + 功能卡片 + 試玩區
  2. Dashboard — 問候語 + stats + 待處理 + 動態 + 推薦摘要
  3. 找工作頁 — tab 切換（AI 推薦 / 我的收藏）+ 篩選
  4. Application 詳情 — tab（面試準備 / 推薦信 / 面試復盤 / 基本資訊）
  5. Kanban 看板 — 5 欄拖拉

- 設計決策紀錄表 → 見 `design-decisions.md`

**面試官想聽你說：**
> 「每個設計決策我都記錄了為什麼。比如推薦和收藏為什麼用 tab 而不是分頁——我先考慮了三個方案，最後選 tab 是因為推薦的唯一動作就是收藏，分開等於多一次跳轉。」

---

### 第六幕：v1 → v2 改版（展示：迭代思維 + 用戶回饋驅動）

**這是最重要的一幕。面試官最想看的不是你做了什麼，而是你「為什麼改」。**

**交付物：**
- v1 vs v2 截圖對比 → 見 `screenshots/`
- 改版決策 Before → Insight → After 表格
- Mockup 驅動開發的流程（5 份 mockup 截圖）

**6 個核心改版故事：**

| 改版 | Before | Insight | After |
|------|--------|---------|-------|
| 爬蟲引擎 | Playwright 被 Cloudflare 擋 | TLS fingerprint 偵測 | Python curl_cffi 偽裝 Chrome |
| 頁面速度 | 3-5 秒跳轉 | `<a>` reload + DB session | `<Link>` + JWT + loading.tsx |
| Dashboard | 只有推薦列表 | 應該是全局總覽 | 問候語+待處理+動態+推薦摘要 |
| 收藏區 | 推薦和收藏分兩頁 | 推薦的唯一動作是收藏 | 合併為 tab 切換 |
| 意圖編輯 | 跳到新頁面 | 編輯完要回原頁面 | 改成 Modal |
| 推薦信 | 泛泛的生成 | 沒用到工作經歷 | WorkExperience + STAR 法則 |

**面試官想聽你說：**
> 「我從這次改版中學到一個方法論：先 mockup 再寫 code。之前寫完整頁 React 才發現佈局不對，現在每個頁面都先寫 HTML mockup 確認方向。」

---

### 第七幕：技術實作 + 衡量 + 反思（展示：技術理解 + 數據思維 + 成長性）

**交付物：**
- 系統架構圖
- DB Schema ERD
- API 端點清單
- 踩坑紀錄表 → 見 `tech-lessons.md`
- 北極星指標 + 淘汰理由
- 三層訊號體系
- 假設驗證計畫（H1-H5）
- 3 個層次的反思（產品/技術/流程）
- Dev Preflight Checklist（把踩坑變成預防機制）

**面試官想聽你說：**
> 「我沒有選 DAU 當北極星，因為登入不代表獲得價值。選的是『每週 AI 面試準備使用次數』——直接反映核心差異化功能有沒有被用到。」

> 「這個專案我最大的收穫不是做了一個 App，而是學到了一套『先想再做』的開發方法論。」

## 面試官最可能問的 5 個問題

1. **「為什麼你覺得這個比直接用 ChatGPT 好？」**
   → 記憶 + 進化 + 全流程整合。ChatGPT 是聰明的陌生人，JobPilot 是跟你一起成長的教練。

2. **「你怎麼決定先做什麼？」**
   → RICE 排序 + 機會成本思維。推薦信 RICE 最高（痛點最大 + 可行性高），AI 進化是差異化但依賴前面的基礎所以放後面。

3. **「如果重來一次你會怎麼做不同？」**
   → 先 mockup 再 code（省掉 70% 的整頁重寫）、Day 1 就設爬蟲監控（不會壞 12 天才知道）、先定義 design tokens（不會逐頁統一顏色）。

4. **「用戶回饋怎麼收集的？」**
   → 自己使用（dogfooding）+ 朋友試用 + 截圖反饋 + 訪談。改版後用 mockup 先確認方向再寫 code。

5. **「這個專案最難的部分是什麼？」**
   → 不是技術，是產品取捨和範圍控制。1 個人 2 個月要做 8 個功能，每個都想做完美，但必須學會砍。

| Roadmap | 橫向時間軸 | 1 頁時間軸 |

| ERD | Mermaid ERD | 不放簡報 |
