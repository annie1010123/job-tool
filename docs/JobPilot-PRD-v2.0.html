1背景與目標

問題陳述

求職者的工具散落在多個平台（104/LinkedIn/Cake/Excel/Notion），每次投遞都在重複「找→改→投→準備→面試」的低效循環。最大的浪費不是時間，是每次面試的經驗沒有被留下來——投了 15 間、面試了 8 間，但第 9 間的準備品質並沒有比第 1 間更好。

目標 Persona

P1 積極求職者：正在找實習/工作的學生和初階工程師，同時投遞 5+ 間公司。

核心 JTBD

當正在積極求職的學生，每天要在多個平台切換找職缺、改履歷、準備面試時，想要一個記得所有資料、會從面試中學習的 AI 助手，以便把時間省下來花在真正重要的面試準備上。

成功指標

5.3 F6 推薦信生成 — 規格

觸發條件

用戶在 Application 詳情頁點擊「推薦信」tab → 選擇語氣 → 點擊「生成推薦信」

正常流程

  1. 前端 POST /api/cover-letter/generate with { applicationId, tone }
  2. 後端讀取用戶 Resume（title, skills, seniority）
  3. 後端讀取 JD（title, companyName, skills, description）
  4. 組裝 prompt → Groq LLM（llama-3.3-70b, temp 0.7）
  5. 回傳推薦信文字（300-500 字）
  6. 前端顯示在可編輯 textarea + 複製按鈕

例外流程

  • 履歷未上傳 → 400 + 「請先上傳履歷」
  • LLM 失敗 → 500 + retry 1 次
  • 回應超時（>15s）→ 前端顯示 timeout 錯誤

5.4 F8 面試錄音復盤 — 規格

觸發條件

用戶在 Application 詳情頁點擊「面試復盤」tab

正常流程（錄音路徑）

  1. 上傳音檔（mp3/m4a/wav, max 50MB）→ Supabase Storage
  2. Whisper API 轉文字（$0.006/min, 中英混合支援）
  3. Groq LLM 萃取 Q&A：每題含 question, answer_summary, quality, improvement_tip, category
  4. 生成 overallFeedback（整體摘要 + 3 改善重點）
  5. 自動寫入 QuestionBank

正常流程（手動路徑）

  1. 用戶填寫多題 Q&A（題目 + 回答 + 自評）
  2. 同樣進入 AI 分析 pipeline（跳過 Whisper）
  3. 自動寫入 QuestionBank

7風險與假設

風險嚴重度預防措施
面試後太累不上傳錄音高推播提醒 + 手動快速問卷 fallback（1 分鐘）
Scope creep 做不完高嚴格 Not Doing List + 每週範圍檢查
104 爬蟲被擋中parse-jd API 可解析單一 URL 作為 fallback
LLM 成本超預算中GPT-4o-mini + Whisper $0.006/min + 每日上限
ChatGPT 加入記憶功能中深耕結構化題庫 + 全流程整合優勢

核心假設驗證計畫

假設驗證方法成功標準
H1: 用戶會上傳錄音追蹤上傳率≥ 30%
H2: AI 進化題目更精準用戶訪談4/5 回答「相關」
H3: 推薦信品質夠好追蹤修改幅度≤ 30% 大幅修改
H4: 語意推薦更精準收藏率≥ 40%
H5: 3 分鐘打動工程主管模擬審閱主動提問技術
