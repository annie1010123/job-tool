# Product Requirements Document：JobPilot — AI 求職教練

> 撰寫視角：開工前（2026-06-09）。本文件以最終交付範圍反推當初的規劃決策，供作品集與 demo 使用。

**作者**：Annie｜**日期**：2026-06-09｜**狀態**：Draft v1.0｜**利害關係人**：開發者本人（兼第一用戶）、未來閱讀此文件的面試官

## 1. Executive Summary

JobPilot 是給台灣求職實習學生的 AI 求職整合工具：把散落在多平台的職缺、投遞進度、推薦信、面試題目與復盤心得收進一站，並讓每一場面試的經驗自動累積成下一場的準備素材。為什麼是現在：本人正處於 PM 實習求職期（同時投遞 5+ 間），既是最痛的用戶，也是能在 2 個月內獨立交付的開發者——用自己的求職過程當真實測試場。

## 2. Background & Context

**問題空間**：積極求職的學生每天在 104、Notion、雲端硬碟、備忘錄之間切換。三個具體痛點：(1) 素材四散，面試前夕拼裝成本高；(2) 同時投遞十幾間，進度全靠記憶；(3) 面試經驗用過即丟，沒有複利。

**市場現況**：職缺推薦（104 本身）、追蹤模板（Notion template）、題庫（PTT/Dcard 考古文）各自存在，**但沒有產品把三者接成迴路**。海外的 Teal/Huntr 做了追蹤，未深入面試準備的累積循環，且不支援台灣職缺來源。

**機會**：差異化不在任何單一功能，在「整合 → 行動 → 累積」的閉環。

## 3. Objectives & Success Metrics

**Goals**
1. 求職素材與流程一站化：從看到職缺到面試復盤，全程不需離開產品
2. 面試準備會累積：第 N 場面試的準備成本顯著低於第 1 場
3. 兩個月內以單人交付可上線 MVP，並以本人求職過程完成 dogfooding

**Non-Goals（明確不做）**
1. 自動投遞——法律風險
2. 面試中即時 AI 提示——道德問題
3. 薪資談判建議（資料不足）、社群功能（不同產品）、付費系統、手機 App（RWD 即可）、多語系、多平台爬蟲（MVP 僅 104＋手動貼 URL）

**Success Metrics**

| 指標 | 現況 | 目標 | 量測方式 |
|------|------|------|----------|
| 北極星：每週 AI 面試準備使用次數 | 0 | ≥ 3 次/週（自用期） | 練習/出題事件計數 |
| 每日推薦點閱率 | — | ≥ 30% | Email/Dashboard 點擊追蹤 |
| 面試復盤完成率 | — | ≥ 70%（每場面試後） | 復盤記錄 / 面試場次 |
| 素材拼裝時間（面試前準備） | 約 2 小時/場 | ≤ 30 分鐘/場 | 自我計時（dogfooding） |

## 4. Target Users & Segments

**主段**：台灣找實習/第一份工作的學生與新鮮人（PM、行銷、UIUX、工程），同時投遞 5+ 間、面試密集。**第一用戶**：開發者本人（PM 實習求職中）——用戶研究、開發、日活使用三位一體。現行替代方案：104 + Notion 模板 + 手寫題庫，切換成本高且不累積。

## 5. User Stories & Requirements

### P0 — Must Have

| # | User Story | 驗收標準 |
|---|-----------|---------|
| 1 | 身為求職者，我要上傳履歷 PDF 並自動拆成經歷素材，讓 AI 之後引用我的真實故事 | 上傳後自動解析成多筆結構化經歷（含 bullets），可編輯 |
| 2 | 我要每天收到與我的履歷及意圖語意相符的職缺推薦 | 每日批次：爬蟲更新 → embedding 比對 → 附相符度與推薦理由 |
| 3 | 我要用看板追蹤所有投遞進度 | 投遞中/面試中/結果三欄；久未回音自動出現在待辦 |
| 4 | 我要為特定職缺一鍵生成客製推薦信 | 履歷 × JD 生成、三種語氣、可編輯保存 |
| 5 | 面試前我要一個備齊所有題目的準備頁 | 作戰室分區：核心題（跨面試共用）／AI 依 JD 出題（行為/技術/動機情境）／該職類歷史考古題 |
| 6 | 我要 AI 教練幫我逐題打磨答案 | AI 初稿、版本管理、評估回饋（30 秒/1 分鐘雙版本） |
| 7 | 面試後我要復盤，且被問的題目自動累積 | 錄音轉文字或手動輸入 → 題目自動回流題庫並標職類 |

### P1 — Should Have

手動貼 URL 收藏任意平台職缺並解析 JD；推薦 Email 每日寄送；職缺依職類自動分類。

### P2 — Nice to Have / Future

推薦門檻數據化調校、跨用戶匿名考古題聚合、多平台爬蟲、面試日曆提醒。

## 6. Solution Overview

**閉環架構**：經歷庫（素材層）→ 推薦與投遞（行動層）→ 作戰室與復盤（累積層），復盤產出回寫素材層形成迴路。

**關鍵技術決策**：Next.js App Router + Prisma + Supabase(pgvector) 單體架構（單人可維護）；LLM 用 Groq Llama 3.3 70B（零成本、低延遲，介面抽象化可換供應商）；embedding 語意比對而非關鍵字（推薦品質是第一印象）；復盤採錄音＋手動雙軌（錄音品質高但上傳率有風險，手動保底）。

## 7. Open Questions

| 問題 | 負責人 | 期限 |
|------|--------|------|
| 面試後太累導致復盤上傳率低怎麼辦？（Pre-mortem 最高風險） | 本人 | Phase 2 前定 fallback 設計 |
| 104 反爬蟲升級的備援？ | 本人 | 保底方案：單一 URL 解析 API |
| 推薦相符度門檻設多少？過嚴沒推薦、過鬆變雜訊 | 本人 | 上線後以分數分布數據決定 |
| Groq 免費層 rate limit 撐得住批次出題嗎？ | 本人 | Phase 1 實測 |

## 8. Timeline & Phasing

| Phase | 期間 | 交付 |
|-------|------|------|
| 1 素材與推薦 | 週 1–3 | 登入、履歷解析＋經歷庫、爬蟲＋embedding 推薦、每日 Email |
| 2 行動層 | 週 4–5 | 職缺收藏、Kanban、推薦信生成 |
| 3 累積層 | 週 6–7 | 作戰室（分區＋AI 教練）、復盤回流題庫 |
| 4 收尾上線 | 週 8 | UI 統一（暖色系）、部署 Vercel、README |

依賴：無外部團隊。風險緩衝：每週對照 Non-Goals 檢查 scope creep。
